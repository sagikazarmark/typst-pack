Throwaway prototype

One lifecycle, two honest adapter surfaces

Question: what concrete CLI commands and Dagger functions expose the settled typst-pack lifecycle without leaking library internals, reviving Resource Slots, or pretending Dagger has terminal and watch capabilities?

Verdict under test

  • Use six CLI commands: create, inspect, compile, watch, materialize, and convert.
  • Keep Typst muscle memory: shared compile and watch flags follow the exact embedded Typst version; Pack-specific identity, override, authority, trust, reporting, and limit controls are named differences.
  • Make advanced creation explicit: flat flags describe one Discovery Variant; repeatable --variant-file values describe a matrix and conflict with the flat variant flags.
  • Use replacement-only overrides: CLI declarations are --override PACK_PATH=FILE; Dagger accepts a project-shaped immutable Directory.
  • Use Dagger objects, not one giant function: typed output specifications and lifecycle result objects wrap native File and Directory values plus machine-readable reports.
  • Narrow honestly: Dagger omits watch, stdout, viewers, color, arbitrary host paths, and Hostile mode. Cache and session-storage topology remains outside this prototype.
Project → create → Pack
Archive or Closure Export → read → Pack
Pack → inspect
Pack → compile or watch → report + artifacts
Pack → materialize → editable project
Pack → convert → archive or closure

Recommended CLI surface

CommandShapeContract
createtypst-pack create [OPTIONS] <INPUT> [OUTPUT]Stabilize one project, run and replay explicit Discovery Variants, then publish an epoch-2 archive.
inspecttypst-pack inspect [OPTIONS] <PACK>Validate archive or Closure Export, then render Pack Inspection as text or JSON without acquisition.
compiletypst-pack compile [OPTIONS] <PACK> [OUTPUT]Prepare one exact request, run one attempt, then deliver the complete artifact collection.
watchtypst-pack watch [OPTIONS] <PACK> [OUTPUT]Own successive Pack-bound sessions, report coverage/currentness, and atomically publish only the latest revision.
materializetypst-pack materialize [OPTIONS] <PACK> <DESTINATION>Publish only baseline project files into an absent destination; no force, merge, dependencies, or overrides.
converttypst-pack convert [OPTIONS] <PACK> <OUTPUT> --to <archive|closure>Validate one representation and publish another without discovery, acquisition, repair, or project materialization.

<PACK> accepts an archive file or Closure Export directory. PACK=- means an archive stream. Archive output may be -; tree output always requires an absent directory. There is no extract --force, generic validate, Resource Slot, or Resource Provider command.

    Dagger schema sketch
    
    
    type TypstPack {
  discoveryVariant(..., overrides: Directory): DiscoveryVariant!
  pdf(...): CompilationOutput!
  png(...): CompilationOutput!
  svg(...): CompilationOutput!
  html(...): CompilationOutput!

  create(
    project: Directory!, input: String! = "main.typ",
    variants: [DiscoveryVariant!], include: [String!]! = [],
    packageDirs: [Directory!]! = [], fontDirs: [Directory!]! = [],
    packageEmbedding: EmbeddingPolicy! = EMBED_ALL,
    fontEmbedding: EmbeddingPolicy! = EMBED_ALL,
    trust: TrustProfile! = PARTIALLY_TRUSTED,
    limits: File
  ): PackCreation!

  readArchive(archive: File!, expectedPackIdentity: String, ...): PackIngress!
  readClosure(closure: Directory!, expectedPackIdentity: String, ...): PackIngress!
}

type Pack {
  identity: String!
  inspect: File!
  archive(epoch: Int! = 2): File!
  closureExport(epoch: Int! = 2): ClosureExport!
  materialize: ProjectMaterialization!
  compile(
    output: CompilationOutput!, overrides: Directory,
    sysInputs: [String!]! = [], features: [Feature!]! = [],
    documentTime: Int, pdfCreationTime: Int,
    packageDirs: [Directory!]! = [], fontDirs: [Directory!]! = [],
    offline: Boolean! = false, limits: File
  ): Compilation!
}

type PackCreation { status: CreationStatus!, report: File!, requirePack: Pack! }
type PackIngress { receipt: File!, pack: Pack! }
type Compilation {
  status: CompilationStatus!
  report: File!
  artifacts: Directory!
  resultIdentity: String
  requireSuccess: Directory!
}
type ClosureExport { tree: Directory!, receipt: File! }
type ProjectMaterialization { project: Directory!, receipt: File! }
    
    
    The exact generated GraphQL names are secondary. The tested idea is the shape: tagged output values reject irrelevant controls before execution; result objects preserve semantic rejection versus operational failure; native Dagger objects carry bytes and trees; requirePack and requireSuccess provide CI-friendly fail-fast accessors without erasing the report.
